# BeautyVoiceTTS 剧本编辑状态统一改造计划

## 背景

当前剧本编辑页面同时存在两套状态机制：

- 一部分操作即时落库：
  - 添加片段
  - 删除片段
  - 单段保存
- 一部分操作只保存在本地草稿：
  - 未点击保存的片段编辑
  - 拖拽排序
  - 顶部“保存剧本”前的整页改动

这会导致以下问题：

- “有未保存改动”状态不可靠
- 添加、删除、修改片段时，行为不一致
- 合成导出页面无法稳定判断哪些片段真正已经修改
- 用户看到的剧本编辑页内容，与后端已保存内容可能不一致

## 改造目标

将剧本编辑页面统一为单一草稿模型：

- 在剧本编辑页面中的任何改动：
  - 添加
  - 删除
  - 修改
  - 排序
- 一律只写入前端草稿
- 只有点击“保存剧本”时，才真正写入后端

同时保证：

- 操作区“有未保存改动 / 已保存”状态语义清晰
- 合成导出页面只反映“已保存后的真实差异”
- 修改片段、添加片段、删除片段、排序片段都采用统一规则

---

## 一、目标行为定义

### 1.1 剧本编辑页面

在剧本编辑页面做任何改动后：

- 页面立即显示 `有未保存改动`
- 当前改动只进入草稿
- 不立即影响后端项目文件

点击 `保存剧本` 后：

- 草稿整体提交到后端
- 页面切换为 `已保存`
- 合成导出页面才能看到对应片段的修改标记

### 1.2 合成导出页面

合成导出页面不读取剧本编辑页的本地草稿。

它只基于：

- 后端已保存的项目脚本
- 项目音频资产
- `stale-report`

进行判定。

因此：

- 未保存的剧本编辑，不应在合成导出页面显示
- 已保存的剧本编辑，才应显示为：
  - `已修改待重新生成`
  - `配置变化待重新生成`
  - `缺失音频`

---

## 二、核心改造原则

### 2.1 单一事实来源

剧本编辑页面应维护两个明确状态源：

- `savedScript`
  - 当前后端已保存版本
- `draftScript`
  - 当前用户编辑中的草稿版本

页面所有交互都只修改 `draftScript`。

### 2.2 保存时机统一

以下操作都不再即时调用后端分段接口：

- 添加片段
- 删除片段
- 修改单段内容
- 调整片段顺序

统一改为：

- 只更新 `draftScript`
- 通过一次 `保存剧本` 提交完整脚本

### 2.3 差异驱动状态

“是否有未保存改动”不再通过零散布尔值拼接判断，而应由：

- `draftScript`
- `savedScript`

之间的差异统一决定。

## 三、前端改造方案

## Phase 1：重构剧本编辑状态模型

### 3.1 ScriptEditorPage 状态结构重构

当前 [ScriptEditorPage.jsx](E:/softs/BeautyVoiceTTS/frontend/src/pages/ScriptEditorPage.jsx) 中已有：

- `drafts`
- `segmentOrder`
- `newSegment`
- `editingId`

建议重构为：

- `savedScript`
- `draftScript`
- `editingId`
- `segmentDraft`

说明：

- `draftScript.segments` 是页面上唯一的片段数据来源
- 新增片段直接写入 `draftScript.segments`
- 删除片段直接从 `draftScript.segments` 中移除
- 排序直接重排 `draftScript.segments`
- 单段编辑保存只更新 `draftScript.segments` 中对应片段

### 3.2 页面渲染规则

片段列表、角色统计、排序、编辑表单都应基于 `draftScript` 渲染。

不要再让：

- 页面部分来自后端 `script`
- 页面部分来自本地 `drafts`

混合叠加。

这样才能保证界面所见即草稿真实状态。

---

## Phase 2：统一所有编辑操作为草稿操作

### 4.1 添加片段

现状：

- 点击“添加片段”会立刻调用后端接口新增片段

目标：

- 只把新片段加入 `draftScript.segments`
- 页面显示 `有未保存改动`
- 不调用后端

### 4.2 删除片段

现状：

- 点击删除会立刻调用后端删除接口

目标：

- 只从 `draftScript.segments` 移除该片段
- 页面显示 `有未保存改动`
- 不调用后端

### 4.3 修改片段

现状：

- 单段保存会调用后端接口更新该段
- 整页保存又是另一套逻辑

目标：

- 单段“保存”改为“应用到草稿”
- 仅更新 `draftScript`
- 页面继续显示 `有未保存改动`
- 不调用后端

### 4.4 排序片段

现状：

- 本地顺序和已保存状态之间容易漂移

目标：

- 排序直接作用于 `draftScript.segments`
- 页面显示 `有未保存改动`
- 不调用后端

---

## Phase 3：统一“保存剧本”行为

### 5.1 保存入口唯一化

真正持久化后端的唯一入口应为：

- 顶部操作区的 `保存剧本`

提交方式：

- 调用 `PUT /projects/{project_id}/script`
- 一次性提交完整 `draftScript`

### 5.2 保存成功后的行为

保存成功后：

- `savedScript = draftScript`
- 页面状态切换为 `已保存`
- 清理编辑中间态
- 刷新当前项目

### 5.3 保存失败后的行为

保存失败后：

- 保留当前草稿不丢失
- 继续显示 `有未保存改动`
- 给出错误提示

---

## 四、差异模型设计

建议新增统一差异工具：

- `frontend/src/utils/scriptDiff.js`

用于比较：

- `savedScript`
- `draftScript`

输出结构建议：

```js
{
  hasChanges: true,
  addedSegmentIds: [],
  removedSegmentIds: [],
  modifiedSegmentIds: [],
  reordered: false
}
```

### 6.1 差异判断维度

片段级差异至少包括：

- `text`
- `speaker`
- `type`
- `emotion`
- `non_verbal`
- `tts_overrides`

结构级差异包括：

- 新增片段
- 删除片段
- 顺序变化

### 6.2 页面状态判定

`有未保存改动` 的条件为：

- `hasChanges === true`

`已保存` 的条件为：

- `hasChanges === false`

这样可以彻底替代当前分散的：

- `hasDraftChanges`
- `hasOrderChanges`

逻辑。

## 五、合成导出页面的配合调整

## Phase 4：合成导出页面只反映“已保存修改”

### 7.1 状态口径

合成导出页面不读取剧本编辑页草稿。

它只读取：

- 后端项目脚本
- 音频资产
- `stale-report`

因此：

- 在剧本编辑页未点击“保存剧本”前
  - 合成导出页面不应显示修改标记
- 点击“保存剧本”后
  - 合成导出页面刷新项目与 `stale-report`
  - 反映真实修改结果

### 7.2 标记语义

保存后，合成导出页面应显示：

- 新增段：`缺失音频`
- 文本修改段：`已修改待重新生成`
- 配置变更段：`配置变化待重新生成`

删除段：

- 不需要在合成页面保留“已删除”行
- 但应影响后续整本重新生成结果

## 六、后端配合改造

## Phase 5：后端判定原因码补强

为支持保存后更准确的重新生成标记，建议在 [tts_routes.py](E:/softs/BeautyVoiceTTS/backend/api/tts_routes.py) 的 `stale-report` 中完善原因码。

建议支持：

- `text_changed`
- `speaker_changed`
- `type_changed`
- `emotion_changed`
- `non_verbal_changed`
- `tts_overrides_changed`
- `missing_audio`
- `voice_assignment_changed`
- `preset_changed`
- `synthesis_config_changed`
- `segment_reordered`

说明：

- 新增片段通常表现为 `missing_audio`
- 删除片段不需要单独在合成页显示
- 如果顺序变化会影响整本时间线，应考虑在整本合成层处理，而不是仅依赖单段状态

## 七、文案与交互统一建议

### 8.1 剧本编辑页文案

建议统一为：

- 单段按钮：`应用到草稿`
- 顶部主按钮：`保存剧本`
- 状态徽标：
  - `有未保存改动`
  - `已保存`

### 8.2 提示文案

当用户执行：

- 添加片段
- 删除片段
- 修改片段

后，统一提示：

- `已加入草稿，点击“保存剧本”后生效`

这样可以清楚传达当前保存语义。

## 八、测试计划

## 9.1 前端测试

建议新增以下测试：

1. 添加片段后显示 `有未保存改动`
2. 删除片段后显示 `有未保存改动`
3. 修改片段后显示 `有未保存改动`
4. 单段“应用到草稿”后仍保持未保存状态
5. 点击 `保存剧本` 后显示 `已保存`
6. `savedScript` 与 `draftScript` 一致时不误报
7. 差异工具可识别：
   - 新增
   - 删除
   - 修改
   - 排序

## 9.2 后端测试

建议补充：

1. 保存新增段后，`stale-report` 返回 `missing_audio`
2. 保存文本修改后，返回 `text_changed`
3. 保存删除后，项目脚本与资产状态一致
4. 保存后合成导出页面使用的项目脚本与后端一致

## 9.3 手工验收

建议按以下链路验证：

1. 新增片段，不保存  
   预期：剧本编辑页显示 `有未保存改动`，合成导出页不变化
2. 删除片段，不保存  
   预期：剧本编辑页显示 `有未保存改动`，合成导出页不变化
3. 修改片段，不保存  
   预期：剧本编辑页显示 `有未保存改动`，合成导出页不变化
4. 点击 `保存剧本`  
   预期：剧本编辑页显示 `已保存`
5. 进入合成导出页  
   预期：
   - 新增段显示 `缺失音频`
   - 修改段显示 `已修改待重新生成`

## 九、推荐实施顺序

建议按以下顺序实施：

1. 重构剧本编辑页为单一草稿模型
2. 增加统一 diff 工具
3. 移除添加、删除、单段保存中的即时落库逻辑
4. 统一 `保存剧本` 为唯一后端写入入口
5. 对接合成导出页面与 `stale-report`
6. 补齐前后端测试

## 十、结论

当前问题不适合继续做局部补丁，而应直接收敛为单一规则：

- 剧本编辑页中的任何改动都只进入草稿
- 只有点击“保存剧本”才落库
- 合成导出页只反映已保存后的真实差异

这套方案能从根本上解决：

- 未保存状态混乱
- 添加/删除/修改行为不一致
- 合成导出页面标记不稳定
