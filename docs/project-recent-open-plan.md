# VoiceLace 项目打开与最近项目显示方案

## 文档状态

- 状态：方案设计完成
- 执行状态：暂不实施
- 适用范围：文本输入页项目下拉、打开项目文件、导入工程 ZIP、最近打开项目恢复

## 一、背景

当前项目已经具备三种与“项目进入”相关的能力：

- 新建项目
- 打开轻量项目文件（`.bvtproject.json`）
- 导入完整工程 ZIP

但目前这三条链路的语义还不够清晰，导致用户在使用时会感到混乱，主要表现为：

1. 从工程 ZIP 导入时，后端会自动在项目名后追加 `(Imported)`。
2. 文本输入页顶部项目下拉框当前显示的是“当前选中项目”，但没有正式定义“最近打开项目”的规则。
3. “打开项目文件”和“导入工程 ZIP”虽然入口不同，但当前前端状态上缺少明确来源区分。
4. 项目文件句柄绑定目前更接近“当前页面状态”，而不是“项目级绑定关系”。

这些问题会引出两个实际使用上的困扰：

- 用户希望项目名保持干净稳定，不希望因为导入来源不同而被系统改名。
- 用户希望下拉框显示的是“最近一次打开/继续编辑的项目”，而不是只看当前数组顺序或临时选中状态。

## 二、目标

本方案目标是统一“项目进入”语义，并明确项目下拉框显示逻辑，使打开体验符合用户直觉。

需要达成的效果：

1. 导入工程 ZIP 时不再自动给项目名追加 `(Imported)`。
2. 文本输入页项目下拉框显示的就是“最近打开的项目”。
3. 应用重启后，仍能自动恢复最近打开项目。
4. `打开项目文件` 与 `导入工程 ZIP` 在数据语义上明确区分。
5. 项目文件回写能力只属于“打开项目文件”链路，不属于“导入工程 ZIP”链路。

## 三、非目标

本方案第一版不包含以下内容：

- 不实现“ZIP 导入覆盖现有项目”
- 不实现“项目文件覆盖当前项目”
- 不实现多窗口/多标签页最近项目同步
- 不实现复杂的“最近打开项目历史列表”页面
- 不在本方案中处理项目重命名冲突的高级交互

## 四、核心设计原则

1. 项目名称应保持用户语义，不因导入方式被系统强行污染。
2. “当前选中项目”和“最近打开项目”必须有明确且稳定的状态定义。
3. “打开项目文件”与“导入工程 ZIP”必须在来源和后续保存行为上严格区分。
4. 最近打开项目的恢复逻辑应稳定、可解释、可测试。
5. 第一版优先保证规则清晰，而不是一次性引入复杂来源管理系统。

## 五、现状问题分析

## 5.1 工程 ZIP 导入改名问题

当前后端 `import_project_archive()` 中存在硬编码逻辑：

- 导入后会执行：
  - `imported_project.name = f"{imported_project.name} (Imported)"`

这会造成：

- 项目名不再等于原始项目名
- 同一个项目多次导入后可读性下降
- 用户在下拉菜单里识别项目时受到噪音干扰

## 5.2 下拉菜单当前并不是真正的“最近打开”

当前文本输入页项目下拉框数据来自：

- `useProjectStore.projects`
- `Select value={currentProject?.id ?? ""}`

实际含义更接近：

- 显示当前内存中 `currentProject`

而不是：

- 显示经过显式定义和持久化的“最近打开项目”

## 5.3 项目进入来源未被显式建模

当前至少存在三种项目进入方式：

- 本地创建
- 打开项目文件
- 导入工程 ZIP

但当前前端项目状态里没有正式的“来源类型”字段，导致：

- 不能稳定判断该项目是否支持“回写项目文件”
- 不能在 UI 或状态恢复逻辑中做出准确区分

## 六、推荐语义定义

## 6.1 打开项目文件

定义：

- 从轻量项目文件恢复一个“继续编辑”的项目副本

行为要求：

- 不改项目名
- 将该项目标记为最近打开项目
- 记录项目文件绑定关系
- 后续“保存项目”优先回写该项目文件

## 6.2 导入工程 ZIP

定义：

- 从完整归档中恢复一个“本地工作副本”

行为要求：

- 不改项目名
- 将该项目标记为最近打开项目
- 不建立项目文件回写绑定
- 后续如需保存项目文件，应走“导出/保存项目文件”逻辑，不能回写 ZIP

## 6.3 新建项目

定义：

- 新建一个本地项目

行为要求：

- 将新项目标记为最近打开项目
- 不带任何项目文件绑定

## 七、推荐状态模型

## 7.1 新增“最近打开项目”状态

建议在前端 `useProjectStore` 中引入：

- `lastOpenedProjectId`

用途：

- 表示最近一次被用户显式打开、切换、导入或创建并进入编辑态的项目

推荐持久化方式：

- `localStorage`

例如：

- `VoiceLace:lastOpenedProjectId`

## 7.2 新增“项目来源类型”状态

建议为当前项目增加来源元信息，至少支持：

- `local`
- `project_file`
- `archive_import`

用途：

- 决定是否应该存在“项目文件回写绑定”
- 决定后续提示文案
- 让调试和问题定位更容易

第一版不一定需要把来源持久化进后端 `project.json`，但前端至少要有稳定的运行时状态。

## 7.3 项目文件绑定改为“按项目保存”

当前状态更接近：

- 全局只有一个 `currentProjectFileHandle`

推荐改为：

- 按项目 ID 保存绑定关系

例如前端 store 中保存：

```json
{
  "projectFileBindings": {
    "project-id-1": {
      "fileName": "xxx.bvtproject.json"
    }
  }
}
```

说明：

- 浏览器的 `FileSystemFileHandle` 本体未必适合完整序列化持久化，但内存与运行时映射应按项目管理，而不是只跟随当前页切换。
- 第一版至少要做到：切换项目时，不错误丢失当前打开项目文件的“可回写身份”。

## 八、后端方案

## 8.1 去掉 ZIP 导入自动改名

在 [project_routes.py](/E:/softs/VoiceLace/backend/api/project_routes.py) 中调整：

- 删除 `imported_project.name = f"{imported_project.name} (Imported)"` 逻辑

改为：

- 保留原始项目名

## 8.2 不依赖项目名区分副本

项目唯一性继续依赖：

- `project.id`

而不是：

- 项目名

因此：

- 即使多个项目同名，也不影响内部正确性

## 8.3 保持导入返回值语义清晰

推荐保留或增强返回值，例如：

- `project_id`
- `project_name`
- `from_project_id`
- `import_source`

其中：

- `import_source = "archive_import"` 或 `import_source = "project_file"`

这样前端不需要靠调用路径硬编码猜测来源。

## 九、前端方案

## 9.1 统一“项目被打开”的动作

建议在 `useProjectStore` 中新增统一动作，例如：

- `markProjectOpened(project, options)`

职责：

- 设置 `currentProject`
- 更新 `lastOpenedProjectId`
- 根据来源更新项目文件绑定
- 调整 `projects` 列表顺序

这样以下操作都走统一入口：

- `createProject`
- `selectProject`
- `importProjectFile`
- `importArchive`

## 9.2 项目列表排序规则

推荐排序规则：

1. 最近打开项目优先显示在第一位
2. 其余项目按 `updated_at` 倒序

这样可保证：

- 下拉框当前值始终对应最近打开项目
- 列表顺序也符合用户直觉

## 9.3 应用启动恢复逻辑

在 [App.jsx](/E:/softs/VoiceLace/frontend/src/App.jsx) 中建议补齐：

1. 先 `loadProjects()`
2. 读取本地 `lastOpenedProjectId`
3. 若该项目存在，则自动 `selectProject(lastOpenedProjectId)`
4. 若不存在，则回退到列表中最新项目
5. 若项目列表为空，则保持空状态

## 9.4 打开项目文件与导入 ZIP 的状态差异

### 打开项目文件

- 设置 `import_source = "project_file"`
- 建立该项目与文件句柄/文件名的绑定
- 将项目标记为最近打开

### 导入工程 ZIP

- 设置 `import_source = "archive_import"`
- 清空该项目的项目文件绑定
- 将项目标记为最近打开

## 9.5 下拉框显示策略

文本输入页第一个下拉框的显示值继续使用：

- `currentProject?.id`

但其语义正式定义为：

- 当前最近打开且当前处于激活状态的项目

这意味着：

- 应用启动后自动恢复最近打开项目
- 导入或打开后立即切换为该项目
- 用户手动切换下拉项目时，也会重写最近打开项目

## 十、UI 与交互建议

## 10.1 名称显示

下拉框主显示内容：

- 只显示 `project.name`

不再显示：

- `(Imported)`

## 10.2 列表辅助信息

若后续需要帮助区分同名项目，建议在下拉列表项中增加次级信息，而不是改主名称。

可选辅助信息：

- 最近更新时间
- 来源标签
  - `项目文件`
  - `ZIP 导入`
  - `本地项目`

第一版可以先不做 UI 增强，只先修正语义。

## 十一、测试计划

## 11.1 后端测试

建议新增或修改以下测试：

1. 工程 ZIP 导入后项目名不追加 `(Imported)`
2. 导入响应中的 `project_name` 等于源项目名

## 11.2 前端状态测试

建议补充：

1. `createProject` 会更新 `lastOpenedProjectId`
2. `selectProject` 会更新 `lastOpenedProjectId`
3. `importProjectFile` 会更新 `lastOpenedProjectId` 并建立文件绑定
4. `importArchive` 会更新 `lastOpenedProjectId` 但不建立文件绑定
5. 切换项目时，项目文件绑定不会错误串到别的项目

## 11.3 启动恢复测试

建议覆盖：

1. 有 `lastOpenedProjectId` 且项目存在时，应用自动恢复该项目
2. `lastOpenedProjectId` 指向已删除项目时，回退到最近更新项目
3. 项目列表为空时不报错

## 11.4 真实验收

至少做一轮手动验收：

1. 导入一个工程 ZIP，确认项目名无 `(Imported)`
2. 关闭并重新打开应用，确认下拉框自动显示该项目
3. 打开一个 `.bvtproject.json`，确认它变成最近打开项目
4. 再切回一个 ZIP 导入项目，确认“保存项目”不再误认为可回写项目文件

- 去掉 ZIP 导入自动追加 `(Imported)`

产出：

- 后端导入项目名称保持原样

## Phase 2：前端最近打开状态引入

## Phase 3：项目来源与绑定关系梳理

目标：

产出：

- 回写逻辑不再混淆

## Phase 4：启动恢复与验收

目标：

- 实现应用启动恢复最近打开项目
- 补齐测试和手动验收

产出：

- 最近打开体验稳定可验收

## 十三、风险与注意事项

1. 去掉 `(Imported)` 后，可能出现多个同名项目。
   这是可接受的，应通过内部 `project.id` 区分，而不是继续污染名称。

2. 若项目文件绑定仍保留为全局字段，会导致切换项目后“保存项目”行为混乱。
   因此文件绑定必须至少具备项目级语义。

3. 启动时若直接恢复最近打开项目而不校验是否存在，会引发空项目错误。
   必须加回退逻辑。

4. 最近打开状态若只更新 `currentProject` 而不持久化，将无法满足重启恢复目标。

## 十四、推荐结论

推荐按以下策略实施：

1. 去掉工程 ZIP 导入项目名中的 `(Imported)`
2. 把“最近打开项目”做成正式状态并持久化
3. 区分 `打开项目文件` 与 `导入工程 ZIP` 的来源语义
4. 将项目文件回写绑定改成项目级，而不是全局当前页级
5. 下拉框显示逻辑以“最近打开项目”为准

这是当前问题下最清晰、最稳定、也最容易验收的一版方案。
