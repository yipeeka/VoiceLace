# BeautyVoiceTTS 项目保存与打开方案

## 文档状态

- 状态：方案设计完成
- 执行状态：暂不实施
- 适用范围：编辑阶段项目保存、轻量项目文件打开、文本输入页入口增强

## 一、背景

当前项目已经具备服务端项目持久化基础：

- 项目对象会以 `project.json` 形式保存在后端项目目录中
- 剧本编辑页已经有“草稿 -> 保存剧本”的页面内机制
- 文本输入页支持创建项目、切换项目、导入完整工程 ZIP

但从用户视角看，仍存在两个明显缺口：

1. 缺少一个明确的“保存项目”动作，无法在任意编辑阶段导出当前项目快照。
2. 文本输入页缺少“打开项目文件”的能力，当前从本地恢复项目主要依赖完整工程 ZIP。

这会带来两个问题：

- 用户在文本输入阶段或剧本编辑阶段，即使已经完成很多工作，也没有一个清晰、轻量、低成本的保存手段。
- 用户如果只是想继续编辑文本、剧本、角色分配和配置，不应该被迫依赖包含音频、字幕、波形等大体积资产的完整工程 ZIP。

## 二、目标

本方案目标是新增“轻量项目文件”的保存与打开能力，使用户可以在任意编辑阶段保存项目，并在文本输入页直接重新打开继续工作。

需要达成的效果：

- 在文本输入页未解析前，也可以保存当前原文内容
- 在剧本编辑页存在未保存草稿时，也可以保存当前草稿状态
- 用户可以从文本输入页直接打开轻量项目文件，而不是只能导入完整工程 ZIP
- 轻量项目文件适合“继续编辑”，完整工程 ZIP 继续承担“完整归档/完整迁移”的职责

## 三、非目标

本方案明确不包含以下内容：

- 不替代完整工程 ZIP 的归档能力
- 不把音频、字幕、波形等大文件打包进轻量项目文件
- 不在本方案中处理历史轻量项目文件迁移逻辑
- 不在本方案中实现“覆盖现有项目”导入模式，第一版默认导入为新项目副本

## 四、核心设计原则

1. 轻量项目文件面向“编辑态恢复”，不是“完整资产归档”
2. 保存项目时必须保存用户当前页面可见的真实草稿，而不是只保存后端当前已落库状态
3. 文本输入页和剧本编辑页的保存语义必须一致
4. “打开项目文件”和“导入完整工程 ZIP”必须在界面和数据语义上明确区分
5. 第一版优先保证稳定、可理解、可验收，不追求一步到位覆盖所有高级合并场景

## 五、推荐方案

推荐新增一种轻量项目文件格式，例如：

- 文件扩展名：`.bvtproject.json`

该文件仅保存继续编辑所需的核心项目状态，不携带完整音频资产。

能力分工如下：

- `保存项目文件`
  - 面向编辑阶段
  - 保存文本、剧本、角色分配、合成参数、必要元数据
- `打开项目文件`
  - 面向继续编辑
  - 从轻量项目文件恢复一个可继续工作的项目
- `导入/导出完整工程 ZIP`
  - 面向完整归档与迁移
  - 继续保留现有入口和语义

## 六、轻量项目文件范围

### 6.1 建议包含

- `schema_version`
- `project.name`
- `project.status`
- `script.title`
- `script.source_text`
- `script.segments`
- `script.characters`
- `script.metadata`
- `voice_assignments`
- `synthesis_config`
- 可选的预设引用快照
  - 只保留必要元信息
  - 不包含参考音频二进制

### 6.2 建议不包含

- `audio_assets` 实际音频产物
- 完整音频文件
- 分段音频文件
- 字幕文件
- 波形 peaks 文件
- 完整工程 ZIP 内的任何二进制资产

### 6.3 导入后的默认语义

导入轻量项目文件后：

- 项目文本、剧本、配置、角色分配恢复
- 音频资产默认视为未携带
- 若用户需要试听或导出，需要重新执行合成

## 七、保存语义定义

本方案最关键的部分，是定义“保存项目”究竟保存什么。

### 7.1 文本输入页

需要保存：

- 当前页面文本框中的 `sourceText`
- 当前自定义提示词（如后续决定纳入项目态）
- 当前选中的项目基础状态

原因：

当前文本输入页中的原文在未触发解析前，主要存在于前端本地状态中。如果保存项目时不显式收集它，用户会误以为“已经保存”，实际重新打开后却丢失原文。

### 7.2 剧本编辑页

需要保存：

- 当前 `draftScript`
- 包括尚未点击“保存剧本”的新增、删除、重排、片段编辑结果

原因：

剧本编辑页已经有“页面草稿”和“已保存剧本”两个层次。如果保存项目文件时只取后端当前剧本，会直接丢掉用户正在编辑的草稿。

### 7.3 语音配置页与合成页

需要保存：

- 当前角色分配
- 当前合成参数
- 当前项目级状态

是否保存页面内未提交的局部编辑草稿，可在实施时根据现有页面状态模型再做细化，但第一版至少要保证“已选配置”能被带走。

## 八、后端设计

### 8.1 新增导出接口

建议新增：

`GET /api/v1/projects/{project_id}/export/project-file`

用途：

- 导出轻量项目文件

职责：

- 从服务端项目对象生成轻量快照
- 清理与音频资产相关的大字段
- 返回 JSON 文件下载

### 8.2 新增导入接口

建议新增：

`POST /api/v1/projects/import/project-file`

用途：

- 从轻量项目文件创建新项目

职责：

- 校验 schema
- 校验项目结构
- 生成新的 `project_id`
- 导入为新项目副本
- 返回新项目信息

### 8.3 导入策略

第一版建议只支持：

- 导入为新项目副本

不建议第一版就支持：

- 覆盖当前项目
- 合并到当前项目

原因：

- 覆盖/合并会引入更多状态冲突
- 容易与现有页面草稿机制冲突
- 验收边界变得不清晰

## 九、前端设计

### 9.1 文本输入页

在 [TextInputPage.jsx](/E:/softs/BeautyVoiceTTS/frontend/src/pages/TextInputPage.jsx) 中新增两个能力：

- `保存项目`
- `打开项目文件`

同时保留现有：

- `导入工程 ZIP`

建议 UI 文案明确区分：

- `打开项目文件`
- `导入工程 ZIP`

避免用户误认为两者等价。

### 9.2 剧本编辑页

在 [ScriptEditorPage.jsx](/E:/softs/BeautyVoiceTTS/frontend/src/pages/ScriptEditorPage.jsx) 中新增：

- `保存项目`

原因：

需求明确指出“可以在任何时间、编辑阶段保存项目”，所以该入口不能只存在于文本输入页。

### 9.3 共享组件

建议抽一个共享组件，例如：

- `ProjectFileActions`

供以下页面复用：

- 文本输入页
- 剧本编辑页

后续如需要，也可扩展到：

- 声音配置页
- 合成导出页

## 十、前端状态组装层

建议新增统一的“项目快照组装层”，不要让每个页面各自手拼导出 payload。

建议新增工具函数，例如：

- `collectEditableProjectState(...)`
- `buildProjectSnapshotPayload(...)`

职责：

- 从 `useProjectStore`
- `useScriptStore`
- 当前页面草稿态

中收集“用户眼下正在编辑的真实状态”，再统一生成项目文件导出 payload。

这样可以避免以下问题：

- 文本输入页保存的是本地原文
- 剧本编辑页保存的是旧剧本
- 两个页面导出的项目文件互相不一致

## 十一、数据一致性要求

实施时必须特别注意以下一致性规则：

1. 保存项目文件时，若剧本编辑页存在未应用的当前片段编辑草稿，需要先合并到导出快照
2. 保存项目文件时，若文本输入页有未解析原文，必须以该原文为准
3. 打开项目文件后，前端 `currentProject`、`script`、`sourceText` 必须同步恢复
4. 打开轻量项目文件后，不应伪造或保留旧音频产物状态

## 十二、建议实施顺序

### Phase 1：定义格式与接口

- 定义 `.bvtproject.json` 结构
- 实现轻量项目导出接口
- 实现轻量项目导入接口
- 增加基础后端测试

### Phase 2：文本输入页接入

- 文本输入页新增“保存项目”“打开项目文件”
- 明确与“导入工程 ZIP”的 UI 区分
- 验证未解析原文可保存并恢复

### Phase 3：剧本编辑页接入

- 剧本编辑页新增“保存项目”
- 导出时带上当前 `draftScript`
- 验证新增/删除/重排/修改草稿可恢复

### Phase 4：统一快照组装层与验收

- 抽象共享的项目快照构建逻辑
- 补齐前端测试
- 完成手工验收链路

## 十三、测试计划

### 13.1 后端测试

建议新增：

- 导出轻量项目文件成功
- 导入轻量项目文件成功
- 未解析状态下 `source_text` 能保存并恢复
- 导入轻量项目文件后不依赖音频资产
- 导入为新项目副本时 `project_id` 正确重建

### 13.2 前端测试

建议新增：

- 文本输入页保存项目时会带上当前原文
- 剧本编辑页保存项目时会带上当前草稿
- 打开项目文件后，项目、剧本、原文同步恢复
- “打开项目文件”和“导入工程 ZIP”入口语义清晰且互不混淆

### 13.3 手工验收

建议手工验收至少覆盖以下链路：

1. 在文本输入页输入原文但不解析，直接保存项目文件
2. 刷新或清空当前状态后，打开该项目文件，确认原文恢复
3. 完成解析并进入剧本编辑页
4. 对剧本做新增、删除、修改、重排，但不点击“保存剧本”
5. 直接点击“保存项目”
6. 刷新或切换项目后，再打开该项目文件
7. 确认所有草稿修改完整恢复

## 十四、验收标准

本方案完成后，至少要满足以下标准：

- 文本输入页仅输入原文、未解析时，也能保存并重新打开项目文件
- 剧本编辑页存在未保存草稿时，也能保存并重新打开项目文件
- 文本输入页可直接打开轻量项目文件，不依赖完整工程 ZIP
- 打开轻量项目文件后可继续编辑，但不会错误显示完整音频资产已恢复

## 十五、风险与注意事项

### 15.1 语义混淆风险

若文案不清晰，用户容易混淆：

- `打开项目文件`
- `导入工程 ZIP`

因此第一版必须在入口名称和说明文字上明确区分两者用途。

### 15.2 草稿丢失风险

若保存项目时只导出服务端当前状态，而不采集当前页面草稿，会造成“用户已保存但内容仍丢失”的严重体验问题。

这是实施时的最高优先级风险。

### 15.3 音频状态误导风险

轻量项目文件不携带音频资产。如果导入后仍显示“可播放/已合成/已导出”等旧状态，会误导用户。

因此导入轻量项目文件后，音频相关状态需要明确重置或重新判定。

## 十六、结论

本方案建议把“项目保存/打开”拆分为两条清晰链路：

- 轻量项目文件：解决随时保存、随时继续编辑
- 完整工程 ZIP：解决完整归档与完整迁移

这是当前 BeautyVoiceTTS 状态模型下最稳妥、最符合用户预期的方案，也能最大程度复用现有项目持久化基础。
