# VoiceLace 项目文件打开语义修复与改进计划

## 文档状态

- 状态：方案已确认，待实施
- 范围：`.bvtproject.json` 打开链路、项目来源识别、项目复用策略、前端最近打开与保存联动
- 约束：本文件只定义修复和改进计划，不包含代码修改

## 一、问题定义

当前系统中，“打开项目文件”实际上仍然更接近“导入一个新的副本”，这与用户直觉不一致。

用户期望的行为是：

- 打开一个项目文件，应当是“继续编辑这个项目”
- 不应每次都生成一个新的本地项目副本
- 不应不断污染项目列表
- 后续保存应优先回写这个项目文件

当前行为带来的主要问题：

1. 同一个 `.bvtproject.json` 反复打开，会不断生成新的 `project.id`
2. 项目列表会快速膨胀，且难以区分哪个才是“正在编辑的那个”
3. “最近打开项目”与“打开的文件”之间没有稳定的一一对应关系
4. “保存项目 / 另存项目 / 打开项目文件”三者语义边界不够清晰

## 二、目标

本次修复的目标是把“打开项目文件”正式定义为：

- 继续编辑同一个逻辑项目

而不是：

- 默认创建一个新的副本

需要达成的结果：

1. 打开同一个项目文件时，优先复用已存在的关联项目
2. 只有在明确没有可复用目标时，才创建新项目
3. 项目文件与本地项目之间建立稳定绑定
4. 保存项目默认回写当前绑定文件
5. 另存项目仍然可以创建一个新的项目文件路径
6. 最近打开项目恢复逻辑与“打开项目文件”语义一致

## 三、非目标

本计划第一阶段不包含以下内容：

- 不实现多人协作冲突合并
- 不实现项目文件历史版本管理
- 不实现多窗口同时编辑同一项目文件的锁机制
- 不实现“强制导入为副本”的复杂分支 UI

## 四、推荐语义

## 4.1 打开项目文件

正式定义为：

- 尝试定位该文件对应的已有本地项目
- 若存在，则恢复并继续编辑该项目
- 若不存在，则首次创建本地项目并建立绑定

因此“打开项目文件”不应默认生成副本。

## 4.2 导入工程 ZIP

继续定义为：

- 从完整工程归档恢复一个本地工作项目

它与“打开项目文件”不同：

- ZIP 导入仍然是导入工作副本
- 不建立项目文件回写绑定

## 4.3 保存项目

定义为：

- 若当前项目绑定了项目文件，则直接回写该文件
- 若当前项目没有绑定项目文件，则进入“另存/选择保存位置”

## 4.4 另存项目

定义为：

- 选择新的项目文件路径写出
- 写出成功后，当前项目切换绑定到这个新文件

这与桌面软件中的“Save As”一致。

## 五、根因分析

当前偏差的根因主要有四类：

## 5.1 后端导入接口语义错误

当前 [project_routes.py](/E:/softs/VoiceLace/backend/api/project_routes.py) 中：

- `/projects/import/project-file`

行为是：

- 读取文件内容
- 构造一个新的 `Project`
- 调用 `save_project(...)`
- 返回新的 `project_id`

这本质上就是“导入副本”。

## 5.2 缺少“项目文件绑定标识”

当前本地项目没有稳定记录以下信息：

- 这个项目是不是从项目文件打开而来
- 它对应哪个项目文件
- 这个项目文件的身份是什么

所以系统无法判断：

- 这次打开的是不是以前已经打开过的同一个文件

## 5.3 前端仅有运行时句柄，没有项目级持久身份

当前前端虽然已经开始建立“项目级文件句柄绑定”，但它仍然主要依赖：

- 浏览器内存中的 `FileSystemFileHandle`
- `localStorage` 中的少量来源信息

这不足以支撑：

- 重启后稳定识别“这是同一个项目文件”

## 5.4 当前“最近打开”状态与“打开文件”不是同一条语义链

最近打开项目目前主要由前端状态维护，而“项目文件打开”的实际复用决策并没有统一收敛到这条语义链上。

结果是：

- 最近打开项目可能是 A
- 当前打开的文件实际上又被导入成了新的 B

这会让用户体验变得混乱。

## 六、推荐设计

## 6.1 新增项目文件身份元数据

建议在后端项目模型中新增一组轻量元数据，建议命名为：

- `project_origin`

推荐结构：

```json
{
  "kind": "local | project_file | archive_import",
  "source_project_id": "optional",
  "project_file_name": "optional",
  "project_file_fingerprint": "optional"
}
```

说明：

- `kind`
  - 本地新建项目：`local`
  - 打开项目文件得到的项目：`project_file`
  - ZIP 导入得到的项目：`archive_import`
- `source_project_id`
  - 来自项目文件内容中的 `source_project_id`
- `project_file_name`
  - 当前绑定文件名，便于调试和 UI 提示
- `project_file_fingerprint`
  - 该项目文件的稳定识别指纹

## 6.2 项目文件指纹设计

推荐为项目文件建立一个稳定的文件身份指纹。

优先方案：

- 对项目文件原始 JSON 内容计算 `sha256`

得到：

- `project_file_fingerprint`

用途：

- 判断某个项目文件是否已经在系统中有对应项目
- 支撑“打开文件时复用已有项目”

注意：

- 这是“文件身份指纹”，不是剧本内容指纹
- 它服务于“同一个文件再次打开”的识别

## 6.3 打开项目文件的复用规则

推荐严格按以下优先级执行：

1. 若当前前端已知该文件句柄绑定到某个本地项目，直接打开该项目
2. 若后端存在 `project_origin.kind == "project_file"` 且 `project_file_fingerprint` 相同的项目，复用该项目
3. 若不存在指纹命中，但存在 `source_project_id` 命中且项目来源为 `project_file`，可作为弱匹配候选
4. 只有以上都失败时，才首次创建本地项目并建立绑定

这条规则意味着：

- “打开项目文件”默认是复用
- “首次打开”才会创建

## 6.4 保存项目与另存项目的行为

### 保存项目

- 有绑定文件：
  - 直接写回原文件
- 无绑定文件：
  - 等同于触发一次另存

### 另存项目

- 选择新的保存位置
- 写入新的 `.bvtproject.json`
- 更新当前项目绑定：
  - `project_file_name`
  - `project_file_fingerprint`
  - 前端句柄绑定

## 七、后端改造计划

## Phase 1：模型与持久化扩展

目标：

- 为项目增加来源与项目文件身份元数据

任务：

1. 扩展 `Project` 模型
2. 为旧项目提供默认值兼容
3. 保证 `save_project/load_project` 无缝读写

产出：

- 后端可以识别本地项目、项目文件项目、ZIP 导入项目

## Phase 2：项目文件指纹能力

目标：

- 为打开与保存链路提供稳定识别依据

任务：

1. 新增文件指纹计算函数
2. 在项目文件导入时计算指纹
3. 在项目文件保存时刷新指纹

产出：

- 可用于复用匹配的 `project_file_fingerprint`

## Phase 3：打开项目文件接口语义修正

目标：

- 把“导入副本”改成“复用或首次创建”

推荐方案：

- 保留现有接口路径 `/projects/import/project-file`
- 但改其内部逻辑语义

新行为：

1. 解析项目文件
2. 计算 `project_file_fingerprint`
3. 查询是否已有匹配项目
4. 若已有匹配项目：
   - 更新其脚本/配置为当前文件内容
   - 返回该项目现有 `project_id`
   - 不创建新项目
5. 若没有匹配项目：
   - 首次创建项目
   - 写入 `project_origin`
   - 返回新 `project_id`

返回值建议新增：

```json
{
  "project_id": "...",
  "project_name": "...",
  "import_source": "project_file",
  "open_mode": "reused | created",
  "project_file_fingerprint": "..."
}
```

## Phase 4：ZIP 导入语义补齐

目标：

- 保持 ZIP 导入为副本语义，但与项目文件打开严格区分

任务：

1. ZIP 导入项目写入 `project_origin.kind = "archive_import"`
2. 不写项目文件指纹
3. 不建立项目文件回写默认关系

产出：

- “打开项目文件”和“导入 ZIP”语义彻底分开

## 八、前端改造计划

## Phase 5：打开项目文件工作流重写

目标：

- 前端正式按“复用或首次创建”的语义处理

任务：

1. `openProjectFileWithPicker()` 读取文件后继续使用现有入口
2. 调用后端接口后，根据返回的 `open_mode`：
   - `reused`：切换到已有项目
   - `created`：进入新建的首次绑定项目
3. 保存当前文件句柄与 `project_id` 的对应关系

产出：

- 用户再次打开同一文件，不会在列表中多出一个新项目

## Phase 6：保存与另存逻辑统一

目标：

- 避免“保存像导出、另存像保存”的混乱

任务：

1. `saveProjectFile()` 支持两种模式：
   - `save`
   - `saveAs`
2. 当前项目若已有绑定句柄，保存直接回写
3. 若无绑定句柄，保存自动进入保存位置选择
4. 另存成功后更新当前项目绑定

产出：

- 保存/另存符合桌面软件直觉

## Phase 7：最近打开项目恢复对齐

目标：

- “最近打开项目恢复”与“打开项目文件”变成同一语义系统

任务：

1. 启动恢复时，不只恢复 `currentProject`
2. 同时恢复：
   - `script`
   - `sourceText`
   - 文件绑定状态
3. 若最近打开的是项目文件项目，则其保存按钮应直接工作

产出：

- 重启后依然像“继续编辑上次打开的文件”

## 九、数据兼容策略

## 9.1 旧项目兼容

旧项目没有 `project_origin` 时：

- 默认视为 `local`

旧项目没有 `project_file_fingerprint` 时：

- 无法参与强匹配复用
- 但不会影响正常使用

## 9.2 首次修复上线后的历史项目文件

当用户第一次用修复后的版本重新打开旧 `.bvtproject.json`：

- 系统会为它补建新的项目文件指纹与绑定

这属于正常升级路径。

## 十、测试计划

## 10.1 后端测试

必须新增：

1. 打开同一个项目文件两次，应返回同一个 `project_id`
2. 打开不同项目文件，应返回不同 `project_id`
3. 首次打开时 `open_mode == "created"`
4. 再次打开时 `open_mode == "reused"`
5. ZIP 导入仍然生成独立项目
6. `project_origin.kind` 持久化正确

## 10.2 前端测试

必须新增：

1. 最近打开项目恢复后，`currentProject` 与 `script/sourceText` 一致
2. 项目文件打开复用已有项目时，不新增列表项
3. 保存项目在已有绑定时直接回写
4. 另存项目会更新绑定
5. ZIP 导入项目不会被错误视为项目文件项目

## 10.3 手动验收

推荐按以下顺序验收：

1. 新建项目，保存为 `A.bvtproject.json`
2. 关闭应用，重新打开 `A.bvtproject.json`
3. 确认未新增副本项目
4. 修改内容后点击保存，确认直接回写 `A.bvtproject.json`
5. 点击另存项目为 `B.bvtproject.json`
6. 确认当前项目后续保存默认写回 `B.bvtproject.json`
7. 导入工程 ZIP，确认仍然是独立本地项目，不影响 `A/B` 文件绑定

## 十一、风险与注意事项

1. 若仅靠文件名匹配而不使用指纹，误判概率较高，不推荐。
2. 若仅靠前端句柄绑定，不足以覆盖重启和跨会话场景。
3. 若“再次打开同一文件”时直接覆盖本地项目内容，必须保证这是预期行为，并在 UI 文案上说明是“继续打开并同步文件内容”。
4. 若后续支持多人编辑或多窗口打开，同一文件的冲突处理需要单独设计，本计划暂不覆盖。

## 十二、推荐结论

推荐按以下原则实施：

1. “打开项目文件”定义为继续编辑，而不是默认复制副本
2. 用 `project_file_fingerprint` 作为项目文件复用主依据
3. 用 `project_origin` 统一表达本地、项目文件、ZIP 导入三类项目
4. 保存默认回写当前绑定文件，另存才创建新路径
5. 最近打开恢复必须同时恢复项目与脚本状态

这是当前最符合用户直觉、最容易长期维护、也最能减少项目列表膨胀问题的一套方案。
