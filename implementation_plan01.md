# BeautyVoiceTTS 详细实施计划 v1

本文档将现有“功能增强建议”细化为可执行实施计划，目标是把当前项目从“可演示的本地 AI 有声书工具”推进为“可持续迭代的专业级生产工具”。文档重点覆盖实施顺序、任务拆分、依赖关系、验收标准、风险和建议排期，便于直接进入开发执行。

## 1. 项目目标

### 1.1 总体目标
- 支持长文本稳定解析，避免 LLM 一次性全文推理导致 OOM 或 Context 超限。
- 支持 TTS 增量合成和缓存复用，缩短重合成耗时。
- 提供更可靠的音频导出、混音、连续试听和错误恢复能力。
- 前后端状态一致、异常可见、可调试、可验收。

### 1.2 非目标
- 暂不引入分布式任务队列。
- 暂不做云端多租户架构。
- 暂不做复杂权限系统或协作编辑。

### 1.3 成功标准
- 10 万字级文本可稳定完成解析，失败率显著下降。
- 对仅修改少量分段的项目，重合成耗时降低到“只处理脏分段”的量级。
- 导出的完整音频、分段音频、字幕和归档包都可稳定生成。
- 前端在 WebSocket 断线、音频加载失败、模型加载失败时不白屏、不挂死。

---

## 2. 当前状态评估

### 2.1 已有基础
- 前端已具备文本输入、剧本审阅、声音配置、合成导出主流程。
- 后端已具备 FastAPI API、WebSocket 进度通道、LLM/TTS/ASR 引擎封装。
- 已接入真实模型路径，具备基础真实推理能力。
- 已有项目持久化、分段结果回显、基础时间线和导出链路。

### 2.2 当前主要短板
- LLM 解析仍偏全文推理，长文本稳定性不足。
- TTS 逐段全量生成，没有脏段识别和缓存复用。
- 完整导出仍偏“拼接式”实现，缺少更专业的混音管线。
- 前端播放与任务容错存在脆弱点，易出现白屏、音频未就绪异常。
- 监控信息不足，显存和任务状态对用户不够透明。

---

## 3. 实施原则

- 优先处理“高价值 + 高频使用 + 容易影响稳定性”的问题。
- 优先做基础能力，再做体验增强。
- 每个阶段都要求“后端能力 + 前端接入 + 验收验证”闭环完成。
- 每个核心能力都要有降级路径，避免单点失败拖垮整个页面或任务。

---

## 4. 阶段划分与优先级

建议按以下顺序执行：

1. Phase 1：核心引擎性能与长文本支持
2. Phase 2：高级混音与导出管线
3. Phase 3：前端交互与体验升级

其中建议再按优先级拆为：

- `P0`: 必须先完成，否则核心流程不稳定
- `P1`: 高价值能力，完成后显著提升体验
- `P2`: 体验增强和可维护性提升

对应关系：

- Phase 1 基本都属于 `P0/P1`
- Phase 2 以 `P1` 为主，部分为 `P0`
- Phase 3 以 `P1/P2` 为主

## 5. Phase 1：核心引擎性能与长文本支持

## 5.1 目标

- 支持长篇文本稳定解析。
- 支持增量 TTS 合成与缓存复用。
- 提供真实的 GPU/VRAM 运行状态监控。

## 5.2 子项 1-A：LLM 分块解析机制

### 功能目标
- 将超长 Source Text 按规则切块，分批送入 LLM。
- 块与块之间保持角色、说话人、描述和命名一致性。
- 最终输出合并后的统一 Script 结构。

### 任务拆分

1. 设计分块策略
- 优先按空行、段落边界切分。
- 超长段落按字符数二次切分，例如 1500-2500 字符区间。
- 为每块保留原文偏移量，便于回溯和调试。

2. 设计上下文传递结构
- 提取并维护 `character_memory`。
- 内容包含：
  - 角色名标准化映射
  - 角色描述摘要
  - 最近出现过的说话模式
  - 可能的叙述者标记

3. 改造 `llm_engine.py`
- 新增 chunked parse 主入口。
- 单块解析后返回：
  - `segments`
  - `characters`
  - `metadata`
  - `memory_delta`
- 将累计 memory 注入下一块 prompt。

4. 合并块结果
- 对角色名做归一化去重。
- 重新计算 segment index。
- 合并 characters appearance_count。
- 保留原始块级 debug 信息供日志查看。

5. 异常和回退
- 单块失败时支持重试。
- 若某块多次失败，可记录错误并停止任务，或按配置启用“跳过失败块”模式。

### 数据结构建议
- 新增 `ChunkParseContext`
- 新增 `CharacterMemoryEntry`
- 新增 `ChunkParseResult`

### 验收标准
- 5 万字、10 万字文本可完成解析。
- 不因全文一次推理导致直接 OOM。
- 同一角色跨块命名不出现明显漂移。
- 最终 Script 可正常进入后续声音配置与合成流程。

### 风险
- 上下文压缩过度会导致角色漂移。
- 块切得过碎会损失语义连续性。

### 建议优先级
- `P0`

---

## 5.3 子项 1-B：TTS 增量合成与缓存

### 功能目标
- 只对发生变化的 segment 重新合成。
- 缓存命中时直接复用历史音频。

### 缓存粒度
- 以 segment 为单位缓存。

### 缓存键建议
- `md5(text + voice_preset_id + voice_mode + ref_audio_path + ref_text + synthesis_config_hash + engine_backend_version)`

### 任务拆分

1. 定义缓存目录结构
- `backend/data/cache/tts/<hash>.wav`
- 可附带 `<hash>.json` 保存元信息。

2. 生成 config hash
- 将影响音频结果的推理参数纳入 hash。
- 包括 `num_step`、`guidance_scale`、`denoise`、`speed`、`gap_duration_ms` 以外的真实推理参数。

3. 改造 `tts_routes.py`
- 每个 segment 合成前先计算 cache key。
- 命中则复制或直接引用缓存文件。
- 未命中则调用真实 TTS 并写入缓存。

4. 增加缓存元数据
- 保存生成时间、音频时长、源文本、预设摘要、模型标识。

5. 脏段识别
- 当前任务开始前对全部 segment 计算缓存命中率。
- 前端显示：
  - 总段数
  - 命中段数
  - 实际生成段数

6. 清理与维护
- 增加缓存统计接口。
- 后续可补充清理策略，例如按大小或最近访问时间回收。

### 验收标准
- 修改单个分段后重新合成，只重新生成对应分段。
- 缓存命中段不再次触发模型推理。
- 导出的完整音频与当前脚本一致。

### 风险
- 漏掉关键参数会导致“错误复用旧音频”。
- 引用克隆音色时，如果 `ref_audio_path` 变了但 hash 不变，会造成缓存污染。

### 建议优先级
- `P0`

---

## 5.4 子项 1-C：真实 GPU 显存监控

### 功能目标
- 将当前假数据状态栏替换为真实设备遥测。
- 让用户能看到模型加载、显存占用和风险预警。

### 任务拆分

1. 后端采集
- 优先用 `torch.cuda` 获取：
  - 是否可用
  - 当前设备
  - 已分配显存
  - 已保留显存
- 若可用，再引入 `pynvml` 获取总显存、利用率、设备名。

2. 遥测服务
- 新增周期性系统状态发布器。
- 通过现有 WebSocket `system-events` 下发。

3. 前端状态栏
- 显示：
  - GPU 名称
  - 总显存 / 已用显存
  - 当前已加载模型
  - 警戒状态

4. 阈值告警
- 高于阈值时显示 warning，例如 85%。
- 接近 OOM 时显示 danger，例如 93%。

### 验收标准
- GPU 机器上显示真实显存数据。
- CPU-only 环境下优雅降级，不报错。
- 模型加载前后显存变化有明显反映。

### 建议优先级
- `P1`

---

## 6. Phase 2：高级混音与导出管线

## 6.1 目标

- 从“字节拼接”升级为“可控后处理管线”。
- 导出完整项目包，而不仅是单个音频文件。
- 让推理参数真正影响底层音频结果。

## 6.2 子项 2-A：专业级 Mixer Engine

### 功能目标
- 统一采样率、通道数、响度。
- 精确插入停顿。
- 减少拼接爆音与音量跳变。

### 任务拆分

1. 新建混音模块
- 建议新增 `backend/engine/mixer_engine.py`

2. 音频标准化
- 所有 segment 先转换到统一格式：
  - mono
  - target sample rate
  - PCM 16-bit 或 float32

3. 响度归一化
- 引入 `pydub` 或后续升级到 `ffmpeg loudnorm`
- 先做基础 RMS/峰值归一化即可

4. 片段拼接
- 插入 `gap_duration_ms`
- 支持可配置 crossfade
- 生成完整音频

5. 为每段记录 timeline
- start_ms
- end_ms
- duration_ms
- speaker

### 验收标准
- 合成后的总音频无明显爆音。
- 不同角色音量差异缩小。
- Timeline 与导出字幕时间能对齐。

### 建议优先级
- `P1`

---

## 6.3 子项 2-B：完整工程打包导出

### 功能目标
- 一键导出完整项目归档，方便保存、迁移和交付。

### 导出内容
- `full.wav` 或 `full.mp3`
- `segments/` 分段音频
- `subtitles/book.srt`
- `subtitles/book.lrc`
- `project.json` 或 `manifest.json`

### 任务拆分

1. 新增 ZIP 导出接口
- 后端临时目录打包后返回下载

2. 生成字幕
- 依据 timeline 生成 `.srt`
- 依据 timeline 生成 `.lrc`

3. 生成 manifest
- 包含项目名、生成时间、模型信息、参数摘要、分段映射

4. 前端导出入口
- 在合成页增加“导出 ZIP”

### 验收标准
- ZIP 可解压。
- 音频、字幕、元信息齐全。
- 字幕时间轴与音频基本一致。

### 建议优先级
- `P1`

---

## 6.4 子项 2-C：OmniVoice 参数深度挂载

### 功能目标
- 当前页面上的合成参数必须真实影响底层推理。

### 任务拆分

1. 梳理前端参数
- `num_step`
- `guidance_scale`
- `denoise`
- 后续可扩展 `speed`、`duration`、`temperature`

2. 梳理后端数据流
- `SynthesisPage`
- `useSynthesisStore`
- `tts_routes.py`
- `tts_engine.py`
- `OmniVoice.generate(**kwargs)`

3. 参数校验
- 后端对范围做限制，避免非法值直接打崩模型。

4. 实验验证
- 同文本不同参数生成结果有可感知差异。

### 验收标准
- 页面修改参数后，生成结果实际变化。
- 参数值会持久化进项目配置。

### 建议优先级
- `P0`

## 7. Phase 3：前端交互与体验升级

## 7.1 目标

- 让试听更流畅。
- 让长任务更可靠。
- 让用户知道系统当前正在做什么、失败在哪里。

## 7.2 子项 3-A：流式连续分段播放

### 功能目标
- 用户可以从任意 segment 开始连续试听后续段落。

### 任务拆分

1. 播放队列模型
- 当前段结束后自动播放下一段
- 支持从时间线或结果列表任意位置开始

2. 播放器状态统一
- 当前播放段
- 队列状态
- 自动前进
- 中断/暂停

3. UI 交互
- 增加“从此处连续播放”
- 增加“停止连续播放”

### 验收标准
- 从第 N 段开始能连续播到最后一段。
- 中途手动切换段落后状态不乱。

### 建议优先级
- `P2`

---

## 7.3 子项 3-B：WebSocket 断线重连与容错

### 功能目标
- 当后端短暂断开或页面网络抖动时，前端能自动恢复。

### 任务拆分

1. WebSocket hook 增强
- 心跳
- 指数退避重连
- 最大重试次数

2. 状态恢复
- 重连后拉取任务当前状态接口
- 补齐中间丢失的任务进度

3. UI 告警
- 显示“连接中断，正在重连”
- 恢复后显示“已恢复连接”

### 验收标准
- 后端重启后页面不永久卡死。
- 长任务执行中断线后，前端可恢复。

### 建议优先级
- `P1`

## 8. 建议开发顺序

建议按下面顺序实施，收益最高、风险最低：

1. `2-C` OmniVoice 参数真实挂载
2. `1-B` TTS 增量缓存
3. `1-A` LLM 分块解析
4. `2-A` Mixer Engine
5. `2-B` ZIP 导出 + 字幕
6. `1-C` GPU 遥测
7. `3-B` WebSocket 容错
8. `3-A` 连续分段播放

原因：
- 先解决“参数看似可调但无效”和“重复生成太慢”这两个核心体验问题。
- 再处理长文本稳定性。
- 然后升级音频后处理与导出。
- 最后做监控和交互打磨。

## 11. 风险与应对

### 风险 1：长文本分块后角色漂移
- 应对：引入角色记忆、名称归一化、块级日志对比

### 风险 2：缓存误命中
- 应对：扩大 hash 影响范围，写入 metadata 便于审计

### 风险 3：音频格式兼容问题
- 应对：统一以 WAV 为中间产物，MP3 作为可选导出

### 风险 4：WebSocket 状态不一致
- 应对：重连后补拉 REST 状态，前端不完全依赖 WS 单次事件

---

## 12. 建议下一步

如果要立刻进入开发，建议直接从以下两个任务开始：

1. `2-C` OmniVoice 参数深度挂载
2. `1-B` TTS 增量合成与缓存

这两个任务对现有主流程收益最大，而且对代码结构改动相对可控，完成后就能明显提升“可用性”和“重复迭代效率”。
